# 비즈니스
이 workflow는 유무선 통신 billing system의 요금 계산 모듈의 설계서를 작성합니다. legacy system이 존재하고 이를 재구축하는 과정입니다.

## 비즈니스 정의
- 무선, 유선, 비회선, 구독상품 등 여러 가지의 상품 유형이 존재합니다.
- 멀티테넌시를 지원합니다.
- 다양한 유형의 요금 항목 (월정액, 통화료, 종량료, 일회성요금, 할인, 부가세, 연체가산금, 선납반제, 자동납부할인, 분리과금)을 계산하여 청구결과를 만들어야 합니다.
- tmforum spec을 따릅니다.


# 기술 정의
## 사용 언어, 기술, dbms
- Java 25
- Spring boot 4.x
- mybatis
- oracle

## 아키텍쳐
- modular monolithic
- 도메인 (11개) -> 컴포넌트 (64개) -> API 순으로 레이어가 구성됩니다.
- 총 64개의 컴포넌트가 존재하고, 요금 계산은 이 중 하나의 컴포넌트에 들어갑니다. 각 컴포넌트는 세 개의 jar로 구성된다고 가정합니다.
  1. 타 컴포넌트에 제공할 인터페이스만 존재하는 jar -> 컴포넌트끼리는 이 단위로 통신합니다.
  2. 1의 구현체가 존재하는 jar -> 아래에서 설명할 bootjar에서 import합니다.
  3. 내 컴포넌트만 쓸 기능이 존재하는 jar
- 각 도메인(billing, order, party 등)을 담당하는 bootjar가 그 도메인에 속한 컴포넌트 jar 구현체들과 타 도메인에 속한 컴포넌트 jar 구현체를 import합니다. 

# 요구사항
1. 요금 계산 케이스가 다양합니다. 케이스가 추가되어도 핵심 비즈니스 로직은 변경하지 않고 데이터를 읽고 쓰는 부분만 strategy pattern을 이용하여 OCP원칙을 준수해야 합니다.
- 정기청구 (대용량 bulk 처리 batch)
- 가입자 별 조회 (실시간 요금 조회, 해지 시 요금 조회 등 oltp)
- 개통 완료 전 예상 요금 조회 (마스터 테이블에는 아직 데이터가 없고 접수 테이블에만 존재)
- 견적 단계 요금 조회 (마스터, 접수 테이블 어디에도 데이터가 없고 상품,할인 기준정보만으로 요금 조회)
- 위 여러가지 케이스를 cover할 수 있는 단일 진입점 api가 필요합니다.
- 정기청구는 스프링배치의 chunk 단위로 수행되어야 하기 때문에 요금계산 api는 여러 건의 계약정보를 받을 수 있어야 합니다. 

2. 요금 계산 중에 수많은 기준정보를 읽어오고 있습니다. legacy system에서는 sql에 마스터 테이블, 기준정보 테이블을 항상 join으로 처리하고 있어서 sql복잡도가 높고 i/o에 비효율도 존재했습니다. 
- 신규 시스템에서는 기준정보를 in memory cache에서 관리합니다. 
- 정기청구 시에는 시작할 때 필요한 기준정보를 모두 memory로 읽어옵니다.
- OLTP 업무에서는 업무 시간 중 기준정보가 변경될 수도 있기 때문에 캐시 무효화, 현행화 정책이 고려되어야 합니다.

3. 여러 요금 항목이 존재하는데 input, output이 유사 유형끼리 그룹핑되어야 합니다.
-----------------------
| 요금항목 | 입력 | 출력 | 특이사항 |
----------------------
| 월정액 | 상품 가입정보, 정지이력, 상품 요금(기준정보) | 사용기간이 from/to로 존재하는 요금 계산 결과 | 일부 특이상품은 추가 정보를 봐야함. 각 선분이력을 박치기하여 구간을 분리해야 함 |
| 일회성 | 가입정보, 기준정보 | 요금 계산 결과(기간 없음) | 종류가 매우 많음. 하지만 하나의 단위로 추상화 필요 |
| 통화료, 종량료 | 가입정보, 기준정보 | 요금 계산 결과(기간 없음) | |
| 할인1 | 위에서 계산한 기간이 존재하는 원금성 항목 계산 결과, 할인 가입정보, 할인 기준정보 | 할인 계산 결과 (기간 존재) |
| 할인2 | 위에서 계산한 기간이 없는 원금성 항목 계산 결과, 할인 가입정보, 할인 기준정보 | 할인 계산 결과 (기간 없음) |
| 연체가산금 | 청구,수납정보, 위에서 계산한 결과 | 계산 결과 (기간 없음) | |
| 자동납부할인 | 청구,수납정보, 위에서 계산한 결과, 자동납부할인 기준정보 | 계산 결과 (기간 없음) | |
| 부가세 | 위에서 계산한 결과, 과세 기준정보 | 계산 결과 (기간 없음) | |
| 선납반제 | 위에서 계산한 결과, 선납내역 | 계산 결과 (기간 없음) | |
| 분리과금 | 위에서 계산한 결과, 분리과금 기준정보 | 계산 결과 (기간 없음) | |

- 할인1이 끝나면 기간 정보를 없앤 group by sum한 정보로 압축해야 합니다. 그래야 할인2를 처리할 수 있습니다.
- 입력 원천은 다양하나 하나의 추상화된 객체를 만들 수 있을 것으로 보입니다.
- 출력은 기간이 있거나 없거나 둘 중 하나입니다.

4. 일부 요금 항목은 처리됐음을 DB에 update해야 합니다.

5. 새로운 요금항목이 추가되어도 근간은 변경되면 안됩니다. 그래서 요금계산 framework와, 이 framework에 끼워넣을 요금항목 구현체가 별도로 존재해야 합니다. 

6. 위 요금항목들을 pipeline, step을 통해 추상화합니다. 여러가지 요금계산 유스케이스가 있기 때문에 요금계산 흐름시 수많은 if/else가 난립하고 있습니다. 테넌트id, 상품유형, 요금계산유스케이스구분에 따라 pipeline을 구성하는 step을 자유자재로 구성할 수 있어야 합니다.
